# AI Agents: Intensive Vibe Coding Capstone Project

> Put your course learning into action by building an agent that solves a real-world problem, helps people, or improves everyday living.

**Status:** Started 2 days ago · Closes in 16 days

---

## Overview

The AI Agents: Intensive Vibe Coding Capstone Project invites participants to build an AI agent using the concepts, tools, and best practices learned in Kaggle's **5-Day AI Agents: Intensive Vibe Coding Course with Google**. Participants will create a practical, real-world solution that demonstrates how AI agents can automate tasks, assist users, solve business challenges, or improve everyday workflows.

Submissions should showcase both the project's value and technical implementation through a **Kaggle Writeup**, **public codebase**, **video demonstration**, and **project link**. Projects may be entered into one of four tracks — *Agents for Good*, *Agents for Business*, *Concierge Agents*, or *Freestyle* — and will be evaluated based on innovation, solution design, communication, and the effective application of course concepts and agent technologies.

## Description

AI agents are rapidly changing how we interact with technology, enabling systems that can reason, take action, and complete complex tasks on behalf of users. In this capstone project, you'll apply the concepts, tools, and techniques learned throughout Kaggle's 5-Day AI Agents: Intensive Vibe Coding Course with Google to build an agent that solves a meaningful real-world problem.

Whether you're creating an assistant that helps individuals stay organized, streamlines business processes, supports social impact initiatives, or explores a completely new idea, this project is an opportunity to move beyond experimentation and develop something useful, practical, and shareable. Participants are encouraged to think creatively, focus on delivering value, and demonstrate how agent-based systems can address real challenges.

Submissions should showcase both the vision behind the project and the technical decisions that bring it to life. Judges will evaluate projects based on problem definition, solution design, implementation quality, effective use of agent technologies, and overall user value. Successful submissions will clearly demonstrate concepts covered in the course while highlighting thoughtful architecture, strong documentation, and a compelling project story.

By the end of the capstone, participants will have more than a prototype — they'll have a portfolio-ready project that demonstrates their ability to design, build, and communicate modern AI agent systems.

## Submission Requirements

Judges will review submissions across four tracks (see **Tracks and Awards** below). While these tracks exist to inspire and focus capstone learners, the judges are always looking for innovative projects with a "wow" factor — winners may be moved between tracks after review if appropriate.

A valid submission must contain:

- Kaggle Writeup
- Media Gallery
- Attached Public Video
- Attached Project Link

The final submission must be made prior to the deadline. **Any un-submitted or draft Writeups by the hackathon deadline will not be considered by the Judges.**

To create a new Writeup, click the "New Writeup" button. After saving the Writeup, a "Submit" button appears in the top right corner.

> **Note:** If you attach a private Kaggle Resource to your public Kaggle Writeup, your private Resource will automatically be made public after the deadline.

### 1. Kaggle Writeup

The Kaggle Writeup serves as your project report. It should include a title, subtitle, and a detailed analysis of your submission. You must select a **Track** for your Writeup in order to submit.

Your Writeup should not exceed **2,500 words**. Submissions over this limit may be subject to penalty.

The following assets must be attached to the Writeup to be eligible:

**a. Media Gallery**
Attach any images and/or videos associated with your submission. A cover image is required to submit your Writeup, and a video is a required part of the submission.

**b. Video**
Attach your video to the Media Gallery. Videos should be 5 minutes or less, and should be published to YouTube.

**c. Public Project Link**
A URL to your working product or interactive demo, so judges can experience your project firsthand. It should be publicly accessible and not require a login or paywall. If a live demo isn't feasible, a link to your public code repository (e.g., GitHub) is required, including detailed setup instructions.

## Evaluation

Submissions must demonstrate at least **three (3)** of the key concepts covered in the course:

| Key Concept | Where to Demonstrate |
|---|---|
| Agent / Multi-agent system (ADK) | Code |
| MCP Server | Code |
| Antigravity | Video |
| Security features | Code or Video |
| Deployability | Video |
| Agent skills (e.g., Agents CLI) | Code or Video |

### Category 1: The Pitch — Problem, Solution, Value (30 points total)

Evaluates the "why" and "what" of the project and how well it's communicated.

| Criteria (points) | Description |
|---|---|
| **Core Concept & Value** (10) | The project's central idea and relevance to the chosen track, focused on innovation and value. Use of agents should be clear, meaningful, and central to the solution. |
| **YouTube Video Submission** (10) | Clarity, conciseness, and quality of messaging within the 5-minute limit. Should articulate, where relevant: Problem Statement, why agents uniquely help solve it, Architecture (images + description), a Demo, and how it was built (tools/technologies used). |
| **Writeup** (10) | How well the written submission articulates the problem, solution, architecture, and the project's journey. |

### Category 2: The Implementation — Architecture, Code (70 points total)

Evaluates the "how" of the project — code quality, technical design, and AI integration.

| Criteria (points) | Description |
|---|---|
| **Technical Implementation** (50) | Quality of solution architecture and code, meaningful use of agents, tool use (especially clever use of existing toolsets). Code should contain comments pertinent to implementation, design, and behaviors. Live deployment isn't required, but if deployed, documentation to reproduce it should be provided. 🚨 **Do not include any API keys or passwords in your code.** |
| **Documentation** (20) | GitHub submissions should include a README.md explaining the problem, solution, architecture, setup instructions, and relevant diagrams or images. |
